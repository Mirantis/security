# Linux Kernel "Copy-Fail" (CVE-2026-31431): Impact on Mirantis Container Runtime (MCR), Mirantis Kubernetes Engine (MKE), and Mirantis Secure Registry (MSR)

### Release Date
2026-05-04

### Overview

On April 29, 2026, a critical vulnerability in the Linux kernel crypto subsystem, dubbed **"Copy-Fail" (CVE-2026-31431)**, was publicly disclosed. This flaw affects kernel 4.14 and newer and allows an unprivileged local user to perform a controlled write into the kernel page cache via the `AF_ALG` socket interface, leading to **Local Privilege Escalation (LPE)**. The vulnerability is listed on the [CISA Known Exploited Vulnerabilities catalog](https://www.cisa.gov/known-exploited-vulnerabilities-catalog?field_cve=CVE-2026-31431).

**A public Proof-of-Concept (PoC) exploit is available.** Mirantis recommends immediate mitigation on all nodes running MCR, MKE, or MSR.

### Affected Products

**Mirantis Container Runtime (MCR) 25.0 and 29** - not vulnerable to CVE-2026-31431. MCR is not itself vulnerable. However, containers running on an unpatched host have access to the vulnerable kernel interface.

**Mirantis Kubernetes Engine (MKE) 3.8 and 3.9** - not vulnerable to CVE-2026-31431. MKE is not itself vulnerable. However, containers running on unpatched MKE nodes have access to the vulnerable kernel interface.

**Mirantis Secure Registry (MSR) 2.9 and 4.x** - not vulnerable to CVE-2026-31431. MSR is not itself vulnerable and does not execute untrusted code. However, MSR runs as containers on Linux hosts, and an unpatched host kernel exposes those hosts to privilege escalation.

### Threat Landscape

#### Risk to MCR Customers

CVE-2026-31431 has two distinct impacts in a containerised environment.

**Privilege escalation within the container**

An unprivileged process inside a container can open an `AF_ALG` socket, `splice()` a readable file (such as a setuid binary like `/usr/bin/su`) into the crypto socket, and overwrite 4 bytes of that file's in-memory page cache entry. The on-disk binary is unmodified. The next process that execs the binary reads the corrupted in-memory version and grants a root shell inside the container. This works in containers started as a non-root user, as the page cache is a shared kernel resource that is not partitioned by Linux namespaces.

**Cross-container page cache contamination via shared image layers**

The kernel page cache is global to the host. In a standard MCR deployment using overlayfs or btrfs, containers that share the same read-only image layers reference the same underlying page cache pages for files in those layers. An attacker in one container can `splice()` a file from a shared image layer and corrupt its page cache entry. All containers on the same host that subsequently exec that binary will execute the attacker's modified version. No access to those other containers is required; the write primitive crosses container boundaries directly.

The in-memory-only nature of the corruption means the on-disk file is unmodified. File integrity monitors that operate on disk state and static image scanners will not detect the corruption. Runtime security tools that monitor syscall activity (such as those that track `AF_ALG` socket creation) may detect the exploit attempt.

The corruption persists until the affected page is evicted from the page cache or the node is rebooted.

**Environments where exposure is elevated**

- Nodes running many containers from shared base images, as the shared lower layers increase the blast radius of a single page cache write.
- Shared CI/CD build nodes executing untrusted code (e.g., open-source dependency builds, user-supplied `Dockerfile` `RUN` steps), where the initial code execution primitive required to launch the exploit is inherent to the workload.

**Required Action:**
- **Patch the Host OS:** Immediately update the Linux kernel on all nodes running MCR to the latest version available from your Linux distribution vendor.

#### Risk to MKE Customers

Containers running on MKE nodes are subject to the same privilege escalation and cross-container page cache contamination risks described above.

**Required Action:**
- **Patch the Host OS:** Immediately update the Linux kernel on all MKE cluster nodes to the latest version available from your Linux distribution vendor.

#### Risk to MSR Customers

MSR is not vulnerable to CVE-2026-31431. However, MSR 2.9 and 4.x run as containers on Linux hosts. If an attacker gains code execution inside an MSR container through a separate vulnerability, an unpatched host kernel could allow them to escalate to root on the physical or virtual host via CVE-2026-31431.

**Required Action:**
- **Patch the Host OS:** Update the Linux kernel on all nodes running MSR to the latest version available from your Linux distribution vendor.

### Mitigations

The permanent fix for CVE-2026-31431 is a patched Linux kernel reverting the vulnerable in-place AEAD optimization. Upstream stable patches are available ([a664bf3d](https://git.kernel.org/stable/c/a664bf3d603dc3bdcf9ae47cc21e0daec706d7a5), [ce42ee42](https://git.kernel.org/stable/c/ce42ee423e58dffa5ec03524054c9d8bfd4f6237), [fafe0fa2](https://git.kernel.org/stable/c/fafe0fa2995a0f7073c1c358d7d3145bcc9aedd8)). Customers should apply kernel updates from their Linux distribution vendor. Vendor advisories for supported distributions:

- [Ubuntu](https://ubuntu.com/blog/copy-fail-vulnerability-fixes-available)
- [SUSE](https://www.suse.com/c/suse-responds-to-the-copy-fail-vulnerability/)
- [Red Hat](https://access.redhat.com/security/cve/cve-2026-31431#cve-details-mitigation)
- [Rocky Linux](https://kb.ciq.com/article/rocky-linux/rl-cve-2026-31431-mitigation)

None of the Mirantis products covered in this advisory use `AF_ALG` sockets and will not be broken by host-level mitigations that block `AF_ALG` socket access.
