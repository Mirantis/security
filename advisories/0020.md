# Guest-to-host escape and host denial of service (Linux kernel "Januscape", CVE-2026-53359) affects Mirantis OpenStack for Kubernetes (MOSK) compute nodes

### Release Date

2026-07-09

### Overview

On July 6, 2026, a high-severity vulnerability in the Linux kernel KVM/x86 hypervisor, dubbed **"Januscape" (CVE-2026-53359)**, was publicly disclosed. The flaw is a use-after-free in the KVM shadow MMU (`kvm_mmu_get_child_sp()` role-reuse logic) that has been present in the kernel since approximately 2010. It affects both Intel (VMX/EPT) and AMD (SVM/NPT) x8664 hosts and can be triggered from a guest virtual machine to corrupt host shadow page state, enabling **guest-to-host escape** (root code execution on the hypervisor) or **host kernel panic (denial of service)**.

**A public Proof-of-Concept (PoC) exploit that triggers host kernel panic is available**; do not run it on production hypervisors. A full guest-to-host escape exploit has not been published. The permanent fix requires a patched host kernel containing upstream fixes for both CVE-2026-53359 and CVE-2026-46113. As of July 9, 2026, Canonical has not yet shipped fixed kernels for any supported Ubuntu release used by MOSK ([Ubuntu CVE tracker](https://ubuntu.com/security/CVE-2026-53359) lists both 22.04 Jammy and 24.04 Noble as **Vulnerable**).

All MOSK compute nodes are x8664 and run QEMU/KVM via libvirt. MOSK enables nested virtualization (`nested=1`) by default on compute nodes by creating `/etc/modprobe.d/qemu-system-x86.conf`. Every compute node therefore runs a vulnerable host kernel until patched.

**Host vs. guest exposure:** the host kernel is vulnerable on all compute nodes. The known guest-to-host attack, however, can only be launched from tenant VMs that receive virtualization CPU extensions (`vmx`/`svm`) and `/dev/kvm` inside the guest. MOSK's default Nova `vcpu_type` is `host-model`, which does not expose these flags. Deployments using `host-passthrough` (or explicit CPU flags for nested-virtualization workloads) are the primary guest-side attack launch points. Mirantis recommends assessing exposure and applying mitigations without delay.

### Affected Products

This bulletin applies to **all versions and releases of Mirantis OpenStack for Kubernetes (MOSK)** on **compute nodes**. Every MOSK compute runs KVM with nested virtualization enabled by default on an unpatched host kernel. The vulnerability is in the Linux kernel KVM subsystem, not in MOSK application code — it is not limited to a single Ubuntu release.

All supported host OS versions are currently without an available Canonical kernel fix for CVE-2026-53359. Mirantis will publish updated host OS images via standard channels once distribution fixes become available.

**Elevated tenant-escape risk** applies to deployments where Nova is configured to pass virtualization extensions to instances — for example `vcpu_type: host-passthrough`, custom CPU models, or per-flavor/per-image CPU overrides that expose `vmx`/`svm`.

Other Mirantis product lines were not evaluated for this advisory.

### Unaffected Products / MOSK Usage Notes

Within MOSK only (no other Mirantis products evaluated for this bulletin):

- **Default** `host-model` **tenant VMs:** predefined libvirt CPU models do not include `vmx`/`svm`; guests typically lack `/dev/kvm`. These VMs **cannot execute the published guest-to-host Januscape PoC**, even though the underlying hypervisor is unpatched.
- `host-passthrough` **tenant VMs:** host CPU features including `vmx`/`svm` are passed through; guests receive `/dev/kvm`. These are the **primary guest-side attack launch points** for Januscape.
- **Management / control-plane nodes** (non-compute): do not host tenant VMs; direct tenant-escape risk close to zero.

### Vulnerability Information

Januscape is a use-after-free in the KVM/x86 shadow MMU. A role confusion in `kvm_mmu_get_child_sp()` causes incorrect shadow page reuse, corrupting rmap accounting. Guest-side actions alone can trigger the bug and dereference freed shadow pages in the host kernel.

### Threat Landscape for MOSK

**Exposure model**


| Layer                   | What is affected                                        | MOSK default                                                                    |
| ----------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **Host kernel**         | Unpatched KVM shadow MMU on compute nodes               | All compute nodes (`nested=1` by default)                                       |
| **Guest attack launch** | Tenant VM with `vmx`/`svm` + `/dev/kvm` + root in guest | **Not** default `host-model`; requires `host-passthrough` or explicit CPU flags |


**Potential Impact (the "Risk")**

If a guest-to-host escape succeeds, the attacker gains host root and effectively breaks tenant isolation. A hypervisor compromise allows an attacker to:

- **Inspect Guest Memory:** Directly dump the RAM of any running VM to extract secrets, private keys, or sensitive customer data.
- **Access Virtual Disks:** Bypass OpenStack's logical volume permissions to read, modify, or corrupt the underlying backing files of any tenant.
- **Intercept Traffic:** Sniff virtual network interfaces (TAP devices) at the host level, intercepting data before it is processed by virtual switches or encapsulated.

**Primary Attack Vectors**

- **Malicious tenant VM with virt flags in guest:** An attacker with root in a VM that has `vmx`/`svm` and `/dev/kvm` can trigger the published Januscape PoC from guest context. Requires Nova configuration beyond default `host-model`.
  - **Probability: High** on compute nodes serving such instances; **Low** for deployments using only default `host-model`.
- **Host kernel DoS via published PoC:** The public PoC is a guest kernel module requiring VMX/NPT in the guest — same preconditions as above.
  - **Probability: High** where virt flags are exposed to tenants; **Low** with default `host-model` only.
- **Host-local trigger** (unprivileged `/dev/kvm` on host): On some Linux distributions `/dev/kvm` is world-accessible; MOSK/Ubuntu host access is normally restricted.
  - **Probability: Low** on MOSK compute nodes under normal hardening.
- **Future undisclosed exploit variants:** Full escape exploit not published.
  - **Probability: Unknown** — patch regardless.

### Mitigations

The permanent fix is installation of host kernel packages containing upstream fixes for CVE-2026-53359 and CVE-2026-46113, delivered through your supported host OS distribution channels. Mirantis will publish updated host OS images via standard channels once Canonical ships patched kernels for the Ubuntu releases in use (22.04 and 24.04); customers should plan rolling updates of their compute nodes accordingly.

### Work arounds

The primary workaround for general compute nodes is a **three-step process**. Run the assessment steps in **(a)** first — especially a `nestedvirt` scan — and relocate nested-virt-dependent workloads to a dedicated hypervisor pool before applying Steps 1–3 on hosts that serve them.

#### (a) Assess exposure

Assessment has three layers: host configuration, instance domain XML (configured exposure), and runtime nested-virt usage (actual usage).

**Layer 1 — Host-side nested KVM** (expected `1` on all MOSK compute nodes):

```shell
# Intel
cat /sys/module/kvm_intel/parameters/nested 2>/dev/null

# AMD
cat /sys/module/kvm_amd/parameters/nested 2>/dev/null
```

**Layer 2 — Instance domain XML (configured exposure):**

The reliable way to determine whether virtualization extensions are configured to be passed through to a VM is to inspect the instance's libvirt domain XML on the hypervisor:

```shell
instance="<instance-name-or-uuid>"
sudo virsh dumpxml "$instance" | grep -Eq "host-passthrough|'vmx'|'svm'"
```

- Exit code **0** (match) → instance is configured to expose virt flags; **elevated Januscape guest-to-host risk**.
- Exit code **1** (no match) → **known guest-to-host PoC not viable** from this instance (host kernel still unpatched).

To audit all instances on a compute node:

```shell
sudo virsh list --uuid | while read -r instance; do
  if sudo virsh dumpxml "$instance" | grep -Eq "host-passthrough|'vmx'|'svm'"; then
    echo "EXPOSED: $instance"
  fi
done
```

On MOSK, libvirt runs in a containerized DaemonSet; run `virsh` from the libvirt pod if not available on the host (e.g. `kubectl -n openstack exec -ti <libvirt-pod> -c libvirt -- virsh ...`).

**Layer 3 — Runtime nested-virtualization usage:**

Before disabling nested virtualization, scan for VMs that have entered nested guest execution using [nestedvirt](https://github.com/vexxhost/nestedvirt) (VEXXHOST). The tool reads KVM `nested_run` debugfs counters; a non-zero value is strong evidence the VM has used nested virtualization during its current lifetime.

```shell
curl -fsSL https://raw.githubusercontent.com/vexxhost/nestedvirt/main/scripts/run-latest.sh | bash
```

Exit codes: **0** = no usage observed; **1** = usage observed (listed VMs may break if nested is disabled); **2** = scan failed.

A zero result means no usage was observed during this scan, not proof the workload will never need nested virt. Scan repeatedly over a representative operational window.


| Layer         | Question answered                      | Tool                                  |
| ------------- | -------------------------------------- | ------------------------------------- |
| Host config   | Is nested enabled on hypervisor?       | `/sys/module/kvm_*/parameters/nested` |
| Domain XML    | Are virt flags configured in instance? | `virsh dumpxml` + grep                |
| Runtime usage | Has any VM actually used nested virt?  | `nestedvirt scan`                     |


**Before applying the workaround:** VMs that actively use nested virtualization (detected by `nestedvirt`, e.g. Kata containers) require **relocation to dedicated hypervisors** where `nested=1` is preserved. Do not apply Steps 1–3 below without evacuating dependent workloads first.

#### (b) Primary workaround 

Apply per compute node during a maintenance window. The goal is to stop passing virtualization CPU flags to guest instances and disable nested KVM on the host. Graceful instance shutdown during hypervisor reboot ensures instances are stopped; when started again after reboot, Nova **regenerates domain XML** without virt flags.

##### Step 1 — Disable passing nested-virtualization CPU flags in OpenStack configuration

Ensure Nova is configured **not** to expose `vmx`/`svm` or `host-passthrough` to instances. Review and revert any of the following:


| Configuration layer          | Location                                                                      | Mitigation                                                       |
| ---------------------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Cluster-wide Nova CPU type   | `OpenStackDeployment` → `spec.features.nova.vcpu_type`                        | Set to `host-model` (default) or remove override                 |
| Per-node-group Nova CPU type | `OpenStackDeployment` → `spec.nodes.<label>.features.nova.vcpu_type`          | Set to `host-model` or remove override                           |
| Custom CPU model             | `OpenStackDeployment` → `spec.nodes.<label>.features.nova.vcpu_type: <MODEL>` | Revert to `host-model` unless verified not to expose `vmx`/`svm` |
| Nova flavor extra spec       | `hw:cpu_mode` on flavor                                                       | Remove or set `hw:cpu_mode=host-model`                           |
| Nova flavor extra spec       | `hw:cpu_model` on flavor                                                      | Remove or set to a model without `vmx`/`svm`                     |
| Glance image metadata        | `hw_cpu_mode` / `hw_cpu_model`                                                | Remove or set `hw_cpu_mode=host-model`                           |


Cluster-wide example:

```yaml
spec:
  features:
    nova:
      vcpu_type: host-model
```

Flavor example:

```shell
openstack flavor set <flavor-name> --property hw:cpu_mode=host-model
```

Running instances retain old domain XML until shut down and started again (Step 3 handles this).

##### Step 2 — Configure the compute node for `nested=0`

In MOSK, Rockoon creates `/etc/modprobe.d/qemu-system-x86.conf` (once, if absent) with `nested=1`. Override the setting with a higher-priority copy of the file:

```shell
cat /etc/modprobe.d/qemu-system-x86.conf
sudo cp /etc/modprobe.d/qemu-system-x86.conf /etc/modprobe.d/zz-qemu-system-x86.conf
sudo sed -i 's/nested=1/nested=0/g' /etc/modprobe.d/zz-qemu-system-x86.conf
```

Do **not** edit the Rockoon-managed original file. The `zz-` prefix ensures the override wins per modprobe lexicographic ordering.

##### Step 3 — Reboot the compute node

The `nested` module parameter is read only on KVM module load. Linux will not reload `kvm_intel`/`kvm_amd` while VMs are running — **host reboot is required**.

Put the node into [maintenance mode](https://docs.mirantis.com/mosk/26.1/ops/general-operations/enable-maintenance/enable-maint-cli.html) before rebooting:

```shell
kubectl patch clusters.cluster.k8s.io -n <projectName> <clusterName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":true}}}}'
kubectl patch machines.cluster.k8s.io -n <projectName> <machineName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":true}}}}'
```

**Instance handling** (before rebooting the target host):


| Instance type                                                            | Action                                                                                                                                                                                                       |
| ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Standard** `host-model` **instances** with no virt flags in domain XML | **Preferred: live-migrate** to a host where `nested=0` is already applied. No CPU flag mismatch applies. **Alternative:** graceful shutdown via host reboot, then `openstack server start` on the same host. |
| **Instances with virt flags** in domain XML                              | **Won't live-migrate** across incompatible CPU flag exposure. Graceful shutdown via host reboot, then `openstack server start` to regenerate domain XML without virt flags.                                  |
| **Nested-virt-dependent workloads** (Kata, etc.)                         | Relocate to dedicated hypervisor pool with `nested=1` preserved; do not apply Steps 1–3 on this host.                                                                                                        |


Reboot the hypervisor. MOSK [graceful instance shutdown](https://docs.mirantis.com/mosk/26.1/ref-arch/cloud-services/compute/graceful-shutdown.html) automatically ACPI-powers off remaining guest instances before the host shuts down. For instances that were shut down (not live-migrated), **start them explicitly** after reboot:

```shell
openstack server start <SERVER-ID>
```

Verify after reboot:

```shell
cat /sys/module/kvm_intel/parameters/nested 2>/dev/null   # expect 0 or N
cat /sys/module/kvm_amd/parameters/nested 2>/dev/null     # expect 0 or N

sudo virsh dumpxml "$instance" | grep -Eq "host-passthrough|'vmx'|'svm'"
# Expect exit code 1 (no match)
```

Exit maintenance mode (machine first, then cluster):

```shell
kubectl patch machines.cluster.k8s.io -n <projectName> <machineName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":false}}}}'
kubectl patch clusters.cluster.k8s.io -n <projectName> <clusterName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":false}}}}'
```

Do **not** run the public Januscape PoC on production hypervisors (triggers host panic).

#### (c) Dedicated hypervisor pool for nested-virt workloads

Workloads that require nested virtualization (Kata containers, KVM-in-VM, etc.) **cannot** go through Steps 1–3 without breaking. Operate them on a **separate compute pool** (host aggregate / availability zone) where `nested=1` is preserved, tenants are isolated, and kernel patching is prioritized. Use `nestedvirt scan` to identify VMs that must remain on this pool.
