# Guest-to-host escape and host denial of service (Linux kernel "Januscape", CVE-2026-53359) affects Mirantis OpenStack for Kubernetes (MOSK) compute nodes

### Release Date

2026-07-09

### Overview

On July 6, 2026, a high-severity vulnerability in the Linux kernel KVM/x86 hypervisor, dubbed **"Januscape" (CVE-2026-53359)**, was publicly disclosed. The flaw is a use-after-free in the KVM shadow MMU (`kvm_mmu_get_child_sp()` role-reuse logic) that has been present in the kernel since approximately 2010. It affects both Intel (VMX/EPT) and AMD (SVM/NPT) x86-64 hosts and can be triggered from a guest virtual machine to corrupt host shadow page state, enabling **guest-to-host escape** (root code execution on the hypervisor) or **host kernel panic (denial of service)**.

**A public Proof-of-Concept (PoC) exploit that triggers host kernel panic is available**; do not run it on production hypervisors. A full guest-to-host escape exploit has not been published. The permanent fix requires a patched host kernel containing upstream fixes for both CVE-2026-53359 and CVE-2026-46113. As of July 9, 2026, Canonical has not yet shipped fixed kernels for any supported Ubuntu release used by MOSK ([Ubuntu CVE tracker](https://ubuntu.com/security/CVE-2026-53359) lists both 22.04 Jammy and 24.04 Noble as **Vulnerable**).

All MOSK compute nodes are x86-64 and run QEMU/KVM via libvirt. MOSK enables nested virtualization (`nested=1`) by default on compute nodes by creating `/etc/modprobe.d/qemu-system-x86.conf`. Every compute node therefore runs a vulnerable host kernel until patched.

On modern CPUs, guest memory normally uses hardware paging (EPT on Intel, NPT on AMD) and the vulnerable shadow MMU is not exercised. **Nested virtualization forces KVM back through the legacy shadow MMU**, which is where the bug lives. If the host CPU supports nested virtualization and nested KVM is enabled on the hypervisor — the MOSK default — **all tenant VMs on that compute node are exposed** to the guest-to-host attack path. An attacker with root inside any tenant VM can trigger the published PoC.

**Treat every MOSK compute node as high-priority** until the host kernel is patched or `nested=0` is applied.

### Affected Products

This bulletin applies to **all versions and releases of Mirantis OpenStack for Kubernetes (MOSK)** on **compute nodes** whose CPUs support nested virtualization (all current MOSK compute hardware) and where nested KVM is enabled — which is the default. The vulnerability is in the Linux kernel KVM subsystem, not in MOSK application code — it is not limited to a single Ubuntu release.

All supported host OS versions are currently without an available Canonical kernel fix for CVE-2026-53359. Mirantis will publish updated host OS images via standard channels once distribution fixes become available.

Other Mirantis product lines were not evaluated for this advisory.

### Unaffected Products / MOSK Usage Notes

Within MOSK only (no other Mirantis products evaluated for this bulletin):

- **Management / control-plane nodes** (non-compute): do not host tenant VMs; direct tenant-escape risk close to zero.

### Vulnerability Information

Januscape is a use-after-free in the KVM/x86 shadow MMU. A role confusion in `kvm_mmu_get_child_sp()` causes incorrect shadow page reuse, corrupting rmap accounting. Guest-side actions alone can trigger the bug and dereference freed shadow pages in the host kernel.

### Threat Landscape for MOSK

**Exposure model**

A MOSK compute deployment is exposed to the guest-to-host attack path when all of the following prerequisites hold:


| Prerequisite | Why it matters | MOSK compute default |
| ------------ | -------------- | -------------------- |
| **Unpatched host kernel** | Contains the vulnerable KVM shadow MMU code | Yes — all compute nodes until Canonical ships fixed kernels |
| **Nested KVM enabled** (`nested=1`) | Forces KVM through the shadow MMU where the bug is triggered; EPT/NPT-only paths are not affected | Yes — Rockoon sets `nested=1` by default |
| **CPU with virtualization extensions** (Intel VMX / AMD SVM) | Required for nested guest execution on the host | Yes — all current MOSK compute hardware |
| **Root inside a tenant VM** | Attacker must load the published PoC kernel module from guest context | Attacker-dependent (standard cloud threat model) |


Prerequisites 1–3 are satisfied on **every MOSK compute node** under default configuration. When all four hold, the guest-to-host attack path is reachable; control-plane nodes are out of scope because they do not host tenant VMs (see Unaffected Products).

**Potential Impact (the "Risk")**

If a guest-to-host escape succeeds, the attacker gains host root and effectively breaks tenant isolation. A hypervisor compromise allows an attacker to:

- **Inspect Guest Memory:** Directly dump the RAM of any running VM to extract secrets, private keys, or sensitive customer data.
- **Access Virtual Disks:** Bypass OpenStack's logical volume permissions to read, modify, or corrupt the underlying backing files of any tenant.
- **Intercept Traffic:** Sniff virtual network interfaces (TAP devices) at the host level, intercepting data before it is processed by virtual switches or encapsulated.

**Primary Attack Vectors**

- **Malicious tenant VM on a compute node with nested KVM enabled:** An attacker with root in any tenant VM can trigger the published Januscape PoC from guest context.
  - **Probability: High** on all MOSK compute nodes under the default configuration.
- **Host kernel DoS via published PoC:** The public PoC is a guest kernel module — same preconditions as above.
  - **Probability: High** on all MOSK compute nodes under the default configuration.
- **Host-local privilege escalation** (unprivileged access to `/dev/kvm` on the host): A separate attack path from guest-to-host escape — an unprivileged local user on the host OS who can open `/dev/kvm` can trigger the same kernel bug directly (without a tenant VM) to escalate to root. On some Linux distributions (e.g. RHEL and derivatives) `/dev/kvm` is world-accessible (`0666`) by default; MOSK compute nodes run Ubuntu, where `/dev/kvm` is restricted to the `kvm` group and this path does not apply under normal hardening.
  - **Probability: Low** on MOSK compute nodes under normal hardening.
- **Future undisclosed exploit variants:** Full escape exploit not published.
  - **Probability: Unknown** — patch regardless.

Treat unexplained KVM host panics that follow guest activity, nested-virt tests, or tenant VM churn as suspicious.

### Mitigations

The permanent fix is installation of host kernel packages containing upstream fixes for CVE-2026-53359 and CVE-2026-46113, delivered through your supported host OS distribution channels. Upstream fixed stable kernels include 6.12.95, 6.6.144, 6.1.177, 5.15.211, and others. Mirantis will publish updated host OS images via standard channels once Canonical ships patched kernels for the Ubuntu releases in use (22.04 and 24.04); customers should plan rolling updates of their compute nodes accordingly.

### Work arounds

Until patched kernels are available, two workarounds can stop virtualization extensions from reaching tenant guests:

1. **Disable nested KVM on the host** (`nested=0`) and reboot the compute node — see **(b)**.
2. **Disable nested-virt CPU flags in Nova** via `cpu_model_extra_flags` and restart each affected instance — see **(c)**. This avoids changing kernel module configuration or rebooting the hypervisor, but must be applied to every instance individually.

Before applying either workaround on a host, scan for workloads that actively depend on nested virtualization and relocate them to a dedicated hypervisor pool (see **(d)** below).

#### (a) Confirm exposure

On each compute node, verify that nested KVM is enabled (expected on all MOSK compute nodes):

```shell
# Intel
cat /sys/module/kvm_intel/parameters/nested 2>/dev/null

# AMD
cat /sys/module/kvm_amd/parameters/nested 2>/dev/null
```

If the value is `1` or `Y` on a CPU that supports nested virtualization, **all tenant VMs on this hypervisor are in scope**.

Disabling nested KVM (`nested=0`, workaround **(b)**) or stripping nested-virt CPU flags from guests (workaround **(c)**) removes the Januscape guest-to-host attack path, but also breaks any tenant workload that runs its own virtual machines inside the guest — for example Kata containers, CI runners that launch KVM-based test VMs, or nested OpenStack/Kubernetes deployments. There is no reliable way to identify these workloads from host configuration alone: a VM may look like any other tenant instance yet depend on nested virt at runtime. Before applying either workaround, scan for VMs that have actually entered nested guest execution using [nestedvirt](https://github.com/vexxhost/nestedvirt) (VEXXHOST). The tool reads KVM `nested_run` debugfs counters on the hypervisor; a non-zero value is strong evidence the VM has used nested virtualization during its current lifetime. Relocate any flagged VMs to a dedicated hypervisor pool (see **(d)**) before applying workaround **(b)** or **(c)** on this host — otherwise those workloads will fail.

```shell
curl -fsSL https://raw.githubusercontent.com/vexxhost/nestedvirt/main/scripts/run-latest.sh | bash
```

Exit codes: **0** = no usage observed; **1** = usage observed (listed VMs may break if nested is disabled); **2** = scan failed.

A zero result means no usage was observed during this scan, not proof the workload will never need nested virt. Scan repeatedly over a representative operational window.

**Before applying either workaround:** VMs that actively use nested virtualization (detected by `nestedvirt`, e.g. Kata containers) require **relocation to dedicated hypervisors** where `nested=1` is preserved. Do not apply **(b)** or **(c)** on hosts serving these workloads without evacuating them first.

#### (b) Workaround: disable nested KVM on the host

Apply per compute node during a maintenance window.

##### Step 1 — Configure the compute node for `nested=0`

In MOSK, Rockoon creates `/etc/modprobe.d/qemu-system-x86.conf` (once, if absent) with `nested=1`. Override the setting with a higher-priority copy of the file:

```shell
cat /etc/modprobe.d/qemu-system-x86.conf
sudo cp /etc/modprobe.d/qemu-system-x86.conf /etc/modprobe.d/zz-qemu-system-x86.conf
sudo sed -i 's/nested=1/nested=0/g' /etc/modprobe.d/zz-qemu-system-x86.conf
```

Do **not** edit the Rockoon-managed original file. The `zz-` prefix ensures the override wins per modprobe lexicographic ordering.

##### Step 2 — Reboot the compute node

The `nested` module parameter is read only on KVM module load. Linux will not reload `kvm_intel`/`kvm_amd` while VMs are running — **host reboot is required**.

Put the node into [maintenance mode](https://docs.mirantis.com/mosk/26.1/ops/general-operations/enable-maintenance/enable-maint-cli.html) before rebooting:

```shell
kubectl patch clusters.cluster.k8s.io -n <projectName> <clusterName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":true}}}}'
kubectl patch machines.cluster.k8s.io -n <projectName> <machineName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":true}}}}'
```

**Instance handling** (before rebooting the target host):


| Instance type                                    | Action                                                                                                                                                                                                       |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Standard tenant instances** on general compute | **Preferred: live-migrate** to a host where `nested=0` is already applied. **Alternative:** graceful shutdown via host reboot, then `openstack server start` on the same host.                               |
| **Nested-virt-dependent workloads** (Kata, etc.) | Relocate to dedicated hypervisor pool with `nested=1` preserved; do not apply Steps 1–2 on this host.                                                                                                        |


Reboot the hypervisor. MOSK [graceful instance shutdown](https://docs.mirantis.com/mosk/26.1/ref-arch/cloud-services/compute/graceful-shutdown.html) automatically ACPI-powers off remaining guest instances before the host shuts down. For instances that were shut down (not live-migrated), **start them explicitly** after reboot:

```shell
openstack server start <SERVER-ID>
```

Verify after reboot:

```shell
cat /sys/module/kvm_intel/parameters/nested 2>/dev/null   # expect 0 or N
cat /sys/module/kvm_amd/parameters/nested 2>/dev/null     # expect 0 or N
```

Exit maintenance mode (machine first, then cluster):

```shell
kubectl patch machines.cluster.k8s.io -n <projectName> <machineName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":false}}}}'
kubectl patch clusters.cluster.k8s.io -n <projectName> <clusterName> \
  --type=merge -p '{"spec":{"providerSpec":{"value":{"maintenance":false}}}}'
```

Do **not** run the public Januscape PoC on production hypervisors (triggers host panic).

#### (c) Workaround: disable nested-virt CPU flags in Nova

This workaround prevents Nova from passing nested-virtualization CPU features into guest domain XML. It does **not** change the host `nested` module parameter — nested KVM remains enabled on the hypervisor, but tenant instances no longer receive the CPU flags needed to exercise the vulnerable shadow-paging path. No compute-node reboot is required; only instance restarts.

Configure `cpu_model_extra_flags` via the low-level `spec:services` section of the `OpenStackDeployment` CR ([MOSK spec:services](https://docs.mirantis.com/mosk/26.1/ref-arch/openstack/openstack-config/osdpl-cr-spec-services.html)). Prefix each flag with `-` to disable it in guest CPU definitions. The exact flag list depends on your host CPU model — **disabling only `vmx` (Intel) or `svm` (AMD) is not sufficient** and will cause instances to fail booting; you must also disable the companion flags that libvirt associates with nested virtualization on your hardware.

To identify the flags to disable, review the libvirt CPU model definitions under `/usr/share/libvirt/cpu_map/` in the libvirt pod, or query host CPU capabilities with `virsh domcapabilities`. Do **not** rely on `virsh dumpxml` — libvirt may pass nested-virtualization flags into guests implicitly and they may not appear as explicit feature entries in the instance domain XML.

Example for AMD EPYC (illustrative — adjust for your CPU model):

```yaml
spec:
  services:
    compute:
      nova:
        values:
          conf:
            nova:
              libvirt:
                cpu_model_extra_flags: -svm,-lbrv,-tsc-scale,-vmcb-clean,-flushbyasid,-pause-filter,-pfthreshold,-vgif
```

After the updated `OpenStackDeployment` is applied, restart each instance on the affected compute nodes so Nova regenerates domain XML with the disabled flags:

```shell
openstack server stop <SERVER-ID>
openstack server start <SERVER-ID>
```

New instances launched after the configuration change pick up the disabled flags automatically. Instances that were not restarted retain their previous domain XML and remain exposed until stopped and started again.

#### (d) Dedicated hypervisor pool for nested-virt workloads

Workloads that require nested virtualization (Kata containers, KVM-in-VM, etc.) **cannot** use the **(b)** or **(c)** workarounds without breaking. Operate them on a **separate compute pool** (host aggregate / availability zone) where `nested=1` is preserved, with strict tenant isolation and kernel patching prioritized. Use `nestedvirt scan` to identify VMs that must remain on this pool. Kernel patching is the only permanent fix for these nodes.
