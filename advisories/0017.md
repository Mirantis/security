# Local privilege escalation (Linux kernel “Dirty Frag”, CVE-2026-43284 and CVE-2026-43500) affects Mirantis OpenStack for Kubernetes (MOSK) cluster nodes when ESP or RxRPC paths are exposed

### Release Date
2026-05-08

### Overview
A critical class of vulnerabilities in the Linux kernel, collectively referred to as **“Dirty Frag”**, was disclosed. The exploit chains two related flaws—**CVE-2026-43284** (IPsec ESP / xfrm receive path) and **CVE-2026-43500** (RxRPC authentication path)—that allow an unprivileged local attacker to manipulate page-cache-backed memory during cryptographic processing, leading to **Local Privilege Escalation (LPE)**.

**A public Proof-of-Concept (PoC) exploit is available.** The underlying issue is distinct from CVE-2026-31431 (“Copy-Fail”), but from an MOSK defender’s perspective the viable attack vectors and blast radius on cluster nodes are the same: any foothold on the host or in a privileged-enough container can be escalated to host root, breaking tenant isolation. Mirantis recommends assessing exposure and applying mitigations without delay.

Vulnerable kernel code on the ESP/xfrm path dates to 2017 and Linux 4.11 onward; MOSK has always shipped—and continues to ship—host OS kernels in that range. **All MOSK versions and releases are therefore covered by this bulletin** (actual exploitability on a node still depends on whether the **esp4**, **esp6**, or **rxrpc** modules are available).

### Affected Products
This bulletin applies to **all versions and releases of Mirantis OpenStack for Kubernetes (MOSK)**. Every MOSK deployment runs nodes on Linux kernels that fall within the affected range disclosed upstream (from 2017 onward for the ESP-related flaw). A node is exposed to privilege escalation via this issue when the **esp4**, **esp6**, or **rxrpc** kernel modules can be loaded or are loaded (including via autoload triggers). Other Mirantis product lines were not evaluated for this advisory.

### Unaffected Products / MOSK Usage Notes
Within MOSK only (no other Mirantis products evaluated for this bulletin):

* **RxRPC (`rxrpc` module):** Mirantis has confirmed that **no standard MOSK component uses RxRPC**, the module is **not loaded by default**, and **disabling `rxrpc` has no known impact on supported MOSK functionality**. Customers should disable `rxrpc` on all nodes in every cluster as part of the workaround below.

* **ESP (`esp4` / `esp6` modules):** These modules are **used only in MOSK clouds that enable IPsec-related functionality**, namely:
  * **East–West tenant traffic encryption** — available for the **OvS** networking backend; **not supported** for **OVN** or **OpenSDN**.
  * **VPN as a Service (VPN aaS)** — supported for **OvS** and **OVN**; **not supported** for **OpenSDN**.

Management clusters and MOSK deployments not using the above features do not require ESP for standard operation and may disable **`esp4`** and **`esp6`** using the workaround below.

### Vulnerability Information
“Dirty Frag” abuses weaknesses where cryptographic processing can occur on page-cache-backed buffers under attacker influence (for example via **`splice()`**-style planting of references), enabling tampering with cached content (for example privileged binaries or sensitive files) without traditional write access to on-disk objects. **CVE-2026-43284** targets the **IPsec ESP** receive path in the kernel’s **xfrm** stack; **CVE-2026-43500** targets the **RxRPC** authentication path.

### Threat Landscape for MOSK

The analysis below aligns with the Copy-Fail (CVE-2026-31431) advisory ([0015](/advisories/0015.md)): the primary attack vectors and probabilities are the same—only the vulnerable kernel interfaces differ (ESP / RxRPC versus AF_ALG / AEAD).

**Potential Impact (the “Risk”)**

If an exploit for these vulnerabilities is successfully executed on a cluster compute node, the resulting Local Privilege Escalation (LPE) effectively breaks tenant isolation. A “host root” compromise allows an attacker to:

* **Inspect Guest Memory:** Directly dump the RAM of any running VM to extract secrets, private keys, or sensitive customer data.
* **Access Virtual Disks:** Bypass OpenStack’s logical volume permissions to read, modify, or corrupt the underlying backing files of any tenant.
* **Intercept Traffic:** Sniff virtual network interfaces (TAP devices) at the host level, intercepting data before it is processed by virtual switches or encapsulated.

**Primary Attack Vectors**

* **Host-Level Access:** An attacker obtaining low-privileged access to the host OS (e.g., via compromised SSH keys or cloud operator credentials) can use this flaw to achieve full root access.
  * **Probability: Low to Medium.** While host access is highly restricted in MOSK, the human element (credential theft or phishing) remains a consistent threat to administrative interfaces.
* **Control / Management Plane Container Compromise:** An attacker compromising a containerized application that comprises the MOSK control or management plane can exploit this to escape the container and take control of the host OS. Public-facing components like the NGINX Ingress controller are high-priority targets; an exploit in the Ingress container provides the foothold needed to run the LPE.
  * **Probability: Medium.** As the primary entry point for cloud APIs, Ingress controllers face constant probing. A zero-day in NGINX or a misconfiguration could provide the necessary initial entry.
* **VM-to-Host Chain:** While cloud users interact only with the *guest OS* kernel, a “Dirty Frag” attack could be the second stage of a breakout chain. An attacker would first need to exploit a hypervisor vulnerability (KVM/QEMU) to reach the host context before escalating to root.
  * **Probability: Very Low.** This requires a sophisticated “chain” of two distinct vulnerabilities. Hypervisor escapes are rare and typically involve high-complexity exploits.

### Mitigations
The permanent fix is **installation of patched kernel modules (and/or kernel packages)** containing upstream fixes for CVE-2026-43284 and CVE-2026-43500, delivered through your supported host OS distribution channels. **Mirantis will publish updated host OS images** via standard channels once fixes are available for supported releases; customers should plan rolling updates of their clusters accordingly.

Mirantis is actively investigating whether IPsec-related MOSK functionality can be retained safely (without disabling `esp4` / `esp6`) through configuration or additional controls until patched modules are available. **This advisory will be updated** if a practical approach is validated.

### Work arounds

Apply the following consistently across nodes (use configuration management at scale). Combine module install rules in a single file if preferred.

#### (a) All nodes, all clusters: disable RxRPC

`rxrpc` is not used by MOSK and is safe to disable everywhere.

1. **Check whether `rxrpc` is loaded:**

```shell
lsmod | grep rxrpc
```

2. **Persistently prevent loading** (use an `install rxrpc /bin/false` rule so the module cannot be loaded on demand):

```shell
echo "install rxrpc /bin/false" | sudo tee /etc/modprobe.d/disable-rxrpc-dirtyfrag.conf
```

3. **If `rxrpc` is loaded:** unload, reboot, or use the emergency cache procedure under **(b)**.

```shell
sudo modprobe -r rxrpc
```

If unload fails because of dependencies, **reboot** during the next maintenance window.

#### (b) Management clusters and MOSK clusters **not** using IPsec-based features: disable ESP

For environments without East–West tenant traffic encryption and without VPN aaS (and for any node roles where ESP is not required), disable `esp4` and `esp6` module loading.

1. **Check whether ESP modules are loaded:**

```shell
lsmod | grep -E '^(esp4|esp6)\s'
```

2. **Persistently prevent loading:**

```shell
printf '%s\n' 'install esp4 /bin/false' 'install esp6 /bin/false' | sudo tee /etc/modprobe.d/disable-esp-dirtyfrag.conf
```

3. **If either module is loaded:**
   * Attempt unload:

     ```shell
     sudo modprobe -r esp4 esp6 rxrpc
     ```

   * **Reboot** the node to ensure a clean kernel state when possible.
   * **Emergency procedure for critical systems** that cannot reboot immediately: drop caches to mitigate stale tampered pages in memory, then **schedule a reboot** as soon as the maintenance window allows:

     ```shell
     echo 3 | sudo tee /proc/sys/vm/drop_caches
     ```

#### (c) MOSK clusters **with** IPsec-based features enabled

For clouds using **East–West tenant traffic encryption** and/or **VPN aaS**, **`esp4` / `esp6` are required** for that functionality. Until patched modules are available, **disable the relevant features using the procedures below** (only the features you use), **or** accept residual exposure until patches arrive—**risk acceptance must be explicit**.

After IPsec-related OpenStack features are turned off and xfrm state is cleared on the nodes, apply **`esp4` / `esp6` blocking and unloading** as described in **(b)**.

**East–West tenant traffic encryption**

1. **Minimal downtime** is expected on tenant networking; however, schedule a **maintenance window**.
2. In the **OpenStackDeployment** custom resource, set **`spec.features.neutron.ipsec.enabled`** to **`false`**:

   ```yaml
   spec:
     features:
       neutron:
         ipsec:
           enabled: false
   ```

3. Wait until the **OpenStackDeploymentStatus** resource reports **`status.osdpl.state`** **`APPLIED`** (reconciliation finished).
4. On **each cluster node**, flush IPsec xfrm state and policy:

   ```shell
   sudo ip xfrm state flush && sudo ip xfrm policy flush
   ```

   Then continue with **`esp4` / `esp6`** disablement and unloading of ESP modules **(b)**:

**VPN aaS**

1. Disabling the VPN aaS extension implies a restart of the **Neutron server** and **L3 agents**, causing downtime for cloud users and for tenant traffic traversing routers. Schedule a **maintenance window**.
2. In the **OpenStackDeployment** custom resource, set **`spec.features.neutron.extensions.vpnaas.enabled`** to **`false`**:

   ```yaml
   spec:
     features:
       neutron:
         extensions:
           vpnaas:
             enabled: false
   ```

3. Wait until the **OpenStackDeploymentStatus** resource reports **`status.osdpl.state`** **`APPLIED`**.
4. On **each cluster node**, flush IPsec xfrm state and policy, then proceed to unloading ESP modules as in **East–West** step **4** above (same `ip xfrm` commands, then **(b)**).

Disabling **both** features requires both CR changes (and the same node-level flush **once** after both are `APPLIED`, unless your operational procedure prefers sequential rollout).

Mirantis will update this advisory if a validated mitigation path emerges that does not require turning off IPsec-related features.

### Verification

Confirm blocked modules cannot be inserted (expect errors similar to):

```shell
sudo modprobe rxrpc
sudo modprobe esp4
sudo modprobe esp6
```

```shell
lsmod | grep -E 'rxrpc|^esp[46]\s'
```

The `grep` commands should return no output when modules are not loaded.
