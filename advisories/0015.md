# Local privilege escalation (Linux kernel “Copy-Fail”, CVE-2026-31431) affects MOSK cluster nodes when AF_ALG/AEAD is exposed

### Release Date
2026-04-29

### Overview
On April 29, 2026, a critical vulnerability in the Linux kernel crypto sub-system, dubbed **"Copy-Fail" (CVE-2026-31431)**, was disclosed. This flaw affects kernels 4.14 and newer. It allows an unprivileged local user to trigger a heap buffer overflow in the algif\_aead module, leading to **Local Privilege Escalation (LPE)**.

**A public Proof-of-Concept (PoC) exploit is currently available.** While the algif\_aead module is present in standard Linux distributions, it is not required for MOSK functionality and is not enabled out of the box. Mirantis recommends immediate mitigation on all cluster nodes.

### Affected Products
This bulletin applies to **Mirantis OpenStack for Kubernetes (MOSK)** deployments: **cluster nodes** running a **Linux kernel version 4.14 or newer** are exposed to CVE-2026-31431 when the **algif\_aead** module can be loaded or is loaded due to a custom configuration applied to the cluster. Other Mirantis product lines were **not** evaluated for this advisory.

### Unaffected Products
Within MOSK only (no other Mirantis products evaluated for this bulletin): Mirantis has confirmed that **no standard component of the MOSK reference architecture** opens AF\_ALG sockets, nor is the algif\_aead module loaded by default in standard MOSK deployments. Standard MOSK cryptographic operations—including TLS termination, IPsec tunneling, and dm-crypt (LUKS) for volume encryption—operate independently of the algif\_aead interface. Disabling this module has no known impact on standard product functionality.

### Vulnerability Information
The "Copy-Fail" vulnerability exists in the algif\_aead kernel module, which provides a user-space interface to Authenticated Encryption with Associated Data (AEAD) algorithms via the AF\_ALG socket family. Due to improper bounds checking during data copying between user and kernel space, an attacker can overwrite kernel memory.

**CVE Identifier** [CVE-2026-31431](https://nvd.nist.gov/vuln/detail/CVE-2026-31431)

**CVSSv3.1** 7.8 HIGH `CVSS:3.1/AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H`

**CWEs** Not specified in this advisory; see the [CVE-2026-31431](https://nvd.nist.gov/vuln/detail/CVE-2026-31431) entry on NVD if CWE identifiers are published there.

### Threat Landscape for MOSK

The following analysis applies only to environments where the algif\_aead module has been manually loaded or triggered via custom configuration.

**Potential Impact (the "Risk")**

If an exploit for this vulnerability is successfully executed on a cluster compute node, the resulting Local Privilege Escalation (LPE) effectively breaks tenant isolation. A "host root" compromise allows an attacker to:

* **Inspect Guest Memory:** Directly dump the RAM of any running VM to extract secrets, private keys, or sensitive customer data.
* **Access Virtual Disks:** Bypass OpenStack's logical volume permissions to read, modify, or corrupt the underlying backing files of any tenant.
* **Intercept Traffic:** Sniff virtual network interfaces (TAP devices) at the host level, intercepting data before it is processed by virtual switches or encapsulated.

**Primary Attack Vectors**

The following scenarios describe how an attacker might gain the initial access required to execute the "Copy-Fail" exploit on a MOSK node, along with an assessment of their probability:

* **Host-Level Access:** An attacker obtaining low-privileged access to the host OS (e.g., via compromised SSH keys or cloud operator credentials) can use this flaw to achieve full root access.
  * **Probability: Low to Medium.** While host access is highly restricted in MOSK, the human element (credential theft or phishing) remains a consistent threat to administrative interfaces.
* **Control / Management Plane Container Compromise:** An attacker compromising a containerized application that comprises the MOSK control or management plane can exploit this to escape the container and take control of the host OS. Public-facing components like the **NGINX Ingress controller** are high-priority targets; an exploit in the Ingress container provides the foothold needed to run the LPE.
  * **Probability: Medium.** As the primary entry point for cloud APIs, Ingress controllers face constant probing. A Zero-Day in NGINX or a misconfiguration could provide the necessary initial entry.
* **VM-to-Host Chain:** While cloud users interact only with the *guest OS* kernel, a "Copy-Fail" attack could be the second stage of a breakout chain. An attacker would first need to exploit a hypervisor vulnerability (KVM/QEMU) to reach the host context before escalating to root.
  * **Probability: Very Low.** This requires a sophisticated "chain" of two distinct vulnerabilities. Hypervisor escapes are rare and typically involve high-complexity exploits.

### Mitigations
The permanent fix requires updating the Linux kernel to a version containing the upstream fix. Once the fix becomes available for the supported versions of Ubuntu, Mirantis will release updated host OS images via the standard channels. Customers should plan a rolling update of their clusters once these resources are available. In the meantime, Mirantis will provide a host OS configuration module to automate the application of the workaround below in large clusters.

### Work arounds
On each node, check whether **algif\_aead** is already loaded; persist the install rule; if the module is not loaded, no further immediate action is required after that; if it is loaded, unload it and reboot for a clean kernel state, or—when reboot cannot happen yet—drop the page cache and schedule a reboot during the next maintenance window. Apply this across **all cluster nodes**; for large clusters, push the changes via a configuration management system (e.g. Ansible) so every node stays consistent.

**Mitigation steps for all cluster nodes:**

1. **Check Current Status:** Determine if the module is currently resident in the kernel memory.

```shell
lsmod | grep algif_aead
```

2. **Persistently Disable Module Loading:** Regardless of current status, prevent the module from being triggered or loaded. Because the algif\_aead module can be auto-loaded on-demand by the kernel when a program opens an AF\_ALG socket, a standard blacklist command is insufficient; use an **`install algif_aead /bin/false`** rule so the module cannot be loaded at all.

```shell
echo "install algif_aead /bin/false" | sudo tee /etc/modprobe.d/disable-algif-aead.conf
```

3. **If the module is NOT loaded:** No further immediate action is required.

4. **If the module IS loaded:**
   * **Unload the module:**

     ```shell
     sudo modprobe -r algif_aead
     ```

   * **Reboot** the node to ensure a clean kernel state.
   * **Emergency Procedure for Critical Systems:** If the node is running critical workloads and cannot be restarted immediately:
     * **Clear Page Cache.** This is a critical defensive step because public exploits for this vulnerability can replace sensitive binaries (like su) in the kernel's page cache. Dropping caches forces the kernel to re-read binaries from the filesystem, purging any potentially compromised versions from memory.

     ```shell
     echo 3 | sudo tee /proc/sys/vm/drop_caches
     ```

     * **Schedule a reboot** as soon as the maintenance window allows to guarantee memory integrity.

**Verification**

To verify the mitigation is active, attempt to load the module manually. It should fail with an "Operation not permitted" or "not found" error:

```shell
# This should return an error if /bin/false is correctly configured
sudo modprobe algif_aead
modprobe: ERROR: ../libkmod/libkmod-module.c:1084 command_do() Error running install command '/bin/false' for module
```

```shell
# This should return no output
lsmod | grep algif_aead
```
