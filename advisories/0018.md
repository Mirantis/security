# Linux Kernel "Dirty Frag" (CVE-2026-43284, CVE-2026-43500): Impact on Mirantis Container Runtime (MCR), Mirantis Kubernetes Engine (MKE), and Mirantis Secure Registry (MSR)

### Release Date

2026-05-11

### Overview

A critical class of vulnerabilities in the Linux kernel, collectively referred to as **“Dirty Frag”**, was disclosed in May 2026. The exploit chains two related flaws—**[CVE-2026-43284][]** (IPsec ESP / xfrm receive path) and **[CVE-2026-43500][]** (RxRPC authentication path)—that allow an unprivileged local attacker to manipulate page-cache-backed memory during cryptographic processing, leading to **Local Privilege Escalation (LPE)**.

**A public Proof-of-Concept (PoC) exploit is available.** The underlying issue is distinct from CVE-2026-31431 (“Copy-Fail”), but the viable attack vectors and blast radius are the same: any foothold on the host or in a privileged-enough container can be escalated toward host root, breaking tenant isolation. Mirantis recommends assessing exposure and applying patches from their operating system vendor without delay.

### Affected Products

**Mirantis Container Runtime (MCR)** — not vulnerable to CVE-2026-43284 or CVE-2026-43500. MCR is not itself vulnerable. However, containers running on an unpatched host use the host kernel. A malicious container or an attacker that gains code execution inside a container through a separate vulnerability could exploit "Dirty Frag" to escalate privileges to root inside the container or compromise other containers on the same host, and ultimately escalate to root on the host by compromising a privileged container. Encrypted overlay container networks depend on the `esp4` and `esp6` kernel modules that are exploited by CVE-2026-43284; mitigating the vulnerability by disabling the aforementioned modules will break encrypted overlay networks as a side effect.

**Mirantis Kubernetes Engine (MKE) 3.8 and 3.9** — not vulnerable to CVE-2026-43284 or CVE-2026-43500 **as MKE application software**. MKE does not ship or replace the Linux kernel. However, **workloads and daemons on MKE nodes use the host kernel**. Nodes running a **vulnerable kernel** with exposed **ESP** or **RxRPC** paths remain at risk of **local privilege escalation** on that node until patched or mitigated according to your Linux vendor. Encrypted overlay container networks depend on the `esp4` and `esp6` kernel modules that are exploited by CVE-2026-43284; mitigating the vulnerability by disabling the aforementioned modules will break encrypted overlay networks as a side effect.

**Mirantis Secure Registry (MSR) 2.9 and 4.x** — not vulnerable to CVE-2026-43284 or CVE-2026-43500 in its own application code. However, MSR instances run as containers on Linux hosts. If an attacker gains code execution inside a MSR container through a separate vulnerability, an unpatched host kernel could allow them to escape the container and escalate to root on the physical or virtual host via the CVE-2026-43284 and CVE-2026-43500 vulnerability chain (collectively known as "Dirty Frag").

### Vulnerability Information
“Dirty Frag” abuses weaknesses where cryptographic processing can occur on page-cache-backed buffers under attacker influence (for example via **`splice()`**-style planting of references), enabling tampering with cached content (for example privileged binaries or sensitive files) without traditional write access to on-disk objects. **[CVE-2026-43284][]** targets the **IPsec ESP** receive path in the kernel’s **xfrm** stack; **[CVE-2026-43500][]** targets the **RxRPC** authentication path.

### Threat Landscape

#### Risk to MCR Customers
CVE-2026-43284 and CVE-2026-43500 have two distinct impacts in a containerised environment.

**Privilege escalation within the container**

Successful exploitation of the vulnerabilities by an unprivileged process inside a container overwrites 4 bytes ([CVE-2026-43284][]) or 8 bytes ([CVE-2026-43500][]) at an attacker-chosen offset within the in-memory page cache for a readable file such as a setuid binary like `/usr/bin/su` or a privileged configuration file like `/etc/passwd`. The on-disk file is unmodified. The next process that execs the binary or reads from the file reads the corrupted in-memory version and grants code execution as root inside the container. This works in containers started as a non-root user, as the page cache is a shared kernel resource that is not partitioned by Linux namespaces.

**Cross-container page cache contamination via shared image layers**

The kernel page cache is global to the host. In a standard MCR deployment using overlayfs or btrfs, containers that share the same read-only image layers reference the same underlying page cache pages for files in those layers. An attacker in one container can `splice()` a file from a shared image layer and corrupt its page cache entry. All containers on the same host that subsequently exec that binary will execute the attacker's modified version. No access to those other containers is required; the write primitive crosses container boundaries directly.

The in-memory-only nature of the corruption means the on-disk file is unmodified. File integrity monitors that operate on disk state and static image scanners will not detect the corruption. The corruption persists until the affected page is evicted from the page cache or the node is rebooted.

**Environments where exposure is elevated**

- Nodes running many containers from shared base images, as the shared lower layers increase the blast radius of a single page cache write.
- Shared CI/CD build nodes executing untrusted code (e.g., open-source dependency builds, user-supplied `Dockerfile` `RUN` steps), where the initial code execution primitive required to launch the exploit is inherent to the workload.

**Required Action:**
- **Patch the host OS:** Immediately update the Linux kernel on all nodes running MCR to the latest version available from your Linux distribution vendor.


#### Risk to MKE Customers

Any **Linux node** that runs MKE components depends on the **host kernel**. Dirty Frag is a **local** attack: an attacker who can execute code on the node as an unprivileged user may be able to escalate to **root on the host** when vulnerable code paths are reachable (notably via **`esp4`**, **`esp6`**, and/or **`rxrpc`** as applicable on that kernel build). A compromised host undermines **confidentiality, integrity, and availability** for all workloads and control-plane containers on that node.

* **Control-plane or privileged container compromise:** A vulnerability or misconfiguration that yields code execution in a **high-privilege** container on a manager may provide the **local** foothold needed to attempt kernel LPE on the **node**. Treat ingress and externally reachable control-plane surfaces as **high-value** hardening targets (i.e. NGINX Ingress controller).

**Required Action:**

- **Patch the host OS:** Update the Linux kernel on **every** MKE cluster node to a version that fixes CVE-2026-43284 and CVE-2026-43500.

#### Risk to MSR Customers

MSR is not vulnerable to [CVE-2026-43284][] or [CVE-2026-43500][] in its own application code. However, MSR instances run as containers on Linux hosts. If an attacker gains code execution inside a MSR container through a separate vulnerability, an unpatched host kernel could allow them to escape the container and escalate to root on the physical or virtual host via the [CVE-2026-43284][] and [CVE-2026-43500][] vulnerability chain (collectively known as "Dirty Frag").

**Required Action:**

- **Patch the Host OS:** Update the Linux kernel on all nodes running MSR to the latest version available from your Linux distribution vendor.

### Mitigations

The **permanent fix** is installation of **patched kernel packages** (and/or patched kernel modules) containing upstream fixes for CVE-2026-43284 and CVE-2026-43500, delivered through your **supported host OS** distribution channels. Plan **rolling** node updates for MKE clusters and any registry or CI hosts running MSR.

Some vendors and researchers have described **temporary** mitigations involving **preventing load** of kernel modules on the vulnerable paths (**`esp4`**, **`esp6`**, **`rxrpc`**). **Trade-offs:**

- Blocking **`rxrpc`** is often **low operational risk** for generic Kubernetes / MKE nodes **if** your distribution and workloads do not require RxRPC (confirm with your vendor and workload owners).
- **`esp4`** and **`esp6`** are required components for IPsec support in the Linux kernel. Blocking those modules has the effect of disabling IPsec, **which is required for MCR/MKE encrypted overlay networks.**
