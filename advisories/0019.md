# Local privilege escalation and sensitive file disclosure (Linux kernel ptrace, CVE-2026-46333) affects Mirantis OpenStack for Kubernetes (MOSK) cluster nodes

### Release Date

2026-06-15

### Overview

On May 15, 2026, a high-severity Linux kernel vulnerability, **[CVE-2026-46333](https://nvd.nist.gov/vuln/detail/CVE-2026-46333)**, was published. A flaw in the kernel's process-tracing (`ptrace`) permission logic lets an unprivileged local user on the host read root-only files (such as `/etc/shadow` or SSH host private keys) and, via related variants, execute commands as root — leading to **credential theft** and **Local Privilege Escalation (LPE)**.

This is primarily a **host-local** attack. A public Proof-of-Concept exists. Exploitation from a standard unprivileged container is unlikely without additional misconfiguration. Mirantis recommends applying the workaround on all MOSK cluster nodes now and patching when fixed host OS images ship.

### Affected Products

**All versions and releases of MOSK.** The flaw is in core ptrace/process-exit logic, requires no optional kernel module, and affects every MOSK node until its kernel is patched. Other Mirantis product lines were not evaluated for this advisory.

### Vulnerability Information

An unprivileged attacker spawns a set-uid-root binary (e.g. `ssh-keysign`, `chage`) and, during a race against the process's exit, attaches to it via `ptrace` and uses `pidfd_getfd()` to steal its open file descriptors and authenticated IPC channels. Because the dumpable check is skipped once the target loses its memory map, YAMA (at the default scope) still permits the parent to attach. The captured handles allow:

* **Disclosure** of root-only files — `/etc/shadow`, SSH host private keys (drives the confidentiality impact).
* **Arbitrary command execution as root** by hijacking privileged D-Bus/systemd connections, e.g. via `pkexec` or `accounts-daemon` (drives the integrity impact).

### Threat Landscape for MOSK

This follows prior MOSK kernel-LPE bulletins ([0015](/advisories/0015.md), [0017](/advisories/0017.md)). Unlike those page-cache flaws, **CVE-2026-46333 does not reliably work from a standard unprivileged container** — the primary attack surface is low-privileged access on the host OS.

**Potential Impact ("the Risk").** Host root on a compute node breaks tenant isolation. An attacker can dump the RAM of any running VM, read or corrupt tenant virtual disks, and sniff host-level TAP devices before traffic reaches the virtual switch.

**Primary Attack Vectors**

* **Host-level access** (the **primary** path): a low-privileged foothold on the host OS — e.g. via stolen SSH keys or operator credentials — can use this flaw to read privileged files and advance toward root. **Probability: Low–Medium**; host access is restricted, but credential theft and phishing remain consistent threats.
* **Privileged-container compromise:** the exploit needs `ptrace` access to host processes that open privileged files, so it does not work from a standard unprivileged container. Public-facing components (e.g. NGINX Ingress) run unprivileged; host-integrated components requiring privileged containers (Open vSwitch, libvirt) are not publicly exposed and sit behind OpenStack APIs/RPC. **Probability: Low.**
* **VM-to-host chain:** a possible later stage after a separate hypervisor/host escape; cloud users touch only the guest kernel. **Probability: Very Low.**

### Mitigations

The permanent fix is patched kernel packages with the upstream fix, delivered via your supported host OS channels. Mirantis will publish updated host OS images for supported releases via standard channels; plan rolling cluster updates accordingly.

### Workarounds

Mirantis has validated that setting kernel **YAMA** `ptrace_scope` to `2` on MOSK cluster nodes blocks the public PoC with no impact on standard MOSK operations. This restricts `ptrace` attachment to processes holding `CAP_SYS_PTRACE` (typically root), breaking the exploit chain. Apply on **every cluster node**.

#### (a) Recommended: apply via `HostOSConfiguration`

Use the [sysctl host OS module](https://github.com/Mirantis/mosk-host-os-modules/tree/main/sysctl) with a `machineSelector` matching **all** management and child-cluster nodes:

```yaml
apiVersion: kaas.mirantis.com/v1alpha1
kind: HostOSConfiguration
metadata:
  name: cve-2026-46333-ptrace-scope
  namespace: default
spec:
  configs:
  - module: sysctl
    moduleVersion: 1.2.0
    values:
      filename: cve-2026-46333-ptrace-scope
      cleanup_before: true
      options:
        kernel.yama.ptrace_scope: "2"
      state: present
  machineSelector:
    matchLabels:
      your-custom-label: "true"
```

See the [module documentation](https://github.com/Mirantis/mosk-host-os-modules/tree/main/sysctl) for supported Ansible versions per MOSK release and for rollback (`state: absent`).

#### (b) Manual per-node procedure

```shell
# 1. Check current value
cat /proc/sys/kernel/yama/ptrace_scope

# 2. Set immediately
echo 2 | sudo tee /proc/sys/kernel/yama/ptrace_scope

# 3. Persist across reboots
echo "kernel.yama.ptrace_scope = 2" | sudo tee /etc/sysctl.d/99-ptrace-scope.conf
sudo sysctl --system
```

**Operational note:** `ptrace_scope = 2` prevents unprivileged users and non-root operators from attaching debuggers to unrelated processes. Mirantis testing found no impact on supported product operations; customers relying on host-level debugging should account for this.
