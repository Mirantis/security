# Local privilege escalation and sensitive file disclosure (Linux kernel ptrace, CVE-2026-46333) affects Mirantis OpenStack for Kubernetes (MOSK) cluster nodes

### Release Date

2026-06-15

### Overview

On May 15, 2026, a high-severity vulnerability in the Linux kernel, **[CVE-2026-46333](https://nvd.nist.gov/vuln/detail/CVE-2026-46333)**, was published. The flaw affects the kernel's process-tracing (`ptrace`) logic and allows an unprivileged local user on the host to read privileged files—such as `/etc/shadow` or SSH host private keys—that are normally accessible only to root. This can lead to **credential theft** and further **Local Privilege Escalation (LPE)** on the node.

**A public Proof-of-Concept (PoC) exploit is available.** Unlike page-cache flaws such as Copy-Fail or Dirty Frag, this is primarily a **host-local attack**: an unprivileged user on the node OS can read host secrets and break tenant isolation. Exploitation from a standard unprivileged container is unlikely without additional misconfiguration. Mirantis recommends immediate mitigation on all MOSK cluster nodes.

### Affected Products

This bulletin applies to **all versions and releases of Mirantis OpenStack for Kubernetes (MOSK)**. Every MOSK deployment runs cluster nodes on Linux kernels that are potentially affected until patched. Unlike kernel flaws that depend on optional modules, CVE-2026-46333 is in core ptrace/process-exit logic and does not require loading a specific kernel module. Other Mirantis product lines were not evaluated for this advisory.

### Vulnerability Information

CVE-2026-46333 is a flaw in the Linux kernel's process-tracing (`ptrace`) permission checks. When privileged programs such as `ssh-keysign` or `chage` open sensitive files and begin shutting down, the kernel briefly leaves those files open while cleaning up the rest of the process. An unprivileged local attacker who times this window correctly can grab the still-open file handles and read root-only content such as `/etc/shadow` or SSH host private keys.

Successful exploitation yields confidentiality impact (reading root-only secrets) and can facilitate further local privilege escalation (for example offline cracking of the root password hash from `/etc/shadow`, or use of stolen SSH host keys for impersonation). Integrity and availability impacts depend on follow-on attacker actions.

### Threat Landscape for MOSK

The analysis below follows the structure of prior MOSK kernel LPE bulletins ([0015](/advisories/0015.md), [0017](/advisories/0017.md)). Unlike those page-cache flaws, **CVE-2026-46333 does not reliably work from a standard unprivileged container**—the primary attack surface is low-privileged access on the host OS.

**Potential Impact (the "Risk")**

If an exploit for CVE-2026-46333 is successfully executed on a cluster compute node, the attacker can read host-level secrets and may escalate toward full host compromise. A "host root" compromise effectively breaks tenant isolation. An attacker with host root can:

- **Inspect Guest Memory:** Directly dump the RAM of any running VM to extract secrets, private keys, or sensitive customer data.
- **Access Virtual Disks:** Bypass OpenStack's logical volume permissions to read, modify, or corrupt the underlying backing files of any tenant.
- **Intercept Traffic:** Sniff virtual network interfaces (TAP devices) at the host level, intercepting data before it is processed by virtual switches or encapsulated.

**Primary Attack Vectors**

- **Host-Level Access:** An attacker obtaining low-privileged access to the host OS (e.g., via compromised SSH keys or cloud operator credentials) can use this flaw to read privileged files and advance toward full root access. This is the **primary** attack path for CVE-2026-46333.
  - **Probability: Low to Medium.** While host access is highly restricted in MOSK, the human element (credential theft or phishing) remains a consistent threat to administrative interfaces.
- **Privileged Control / Management Plane Container Compromise:** Unlike Copy-Fail or Dirty Frag, this flaw does not work from a standard unprivileged container. The exploit requires `ptrace` access to host processes that open privileged files. In MOSK, public-facing components such as the NGINX Ingress controller run as unprivileged containers and therefore do not provide a direct path to this exploit. Components of the control and management plane that are deeply intergrated with the host OS and therefore require privileged containers run (for example Open vSwitch, libvirt) are not exposed to public; they sit behind OpenStack APIs and RPC mechanisms.
  - **Probability: Low.** Reaching host secrets via this CVE through a container foothold would require compromise of a privileged workload and further misconfiguration—both unlikely in the MOSK reference architecture.
- **VM-to-Host Chain:** A ptrace-based attack could be a later stage of a breakout chain after a hypervisor or host-context primitive. Cloud users interact only with the *guest OS* kernel; reaching the host still requires a prior escape or host foothold.
  - **Probability: Very Low.** This requires a sophisticated chain of distinct vulnerabilities.

### Mitigations

The permanent fix is installation of patched kernel packages containing the upstream fix for CVE-2026-46333, delivered through your supported host OS distribution channels. Mirantis will publish updated host OS images via standard channels once fixes are available for supported releases; customers should plan rolling updates of their clusters accordingly.

### Work arounds

Mirantis has validated that setting the kernel **YAMA** `**ptrace_scope`** to `**2**` on MOSK cluster nodes blocks the public PoC and does not impair standard MOSK functionality. This setting restricts `ptrace` so that only processes holding `CAP_SYS_PTRACE` (typically root) can attach to arbitrary processes; unprivileged local users cannot perform the `ptrace` operations required by the exploit chain.

Apply on **every cluster node**. For large clusters, use the MOSK day-2 `HostOSConfiguration` API with the [sysctl host OS module](https://github.com/Mirantis/mosk-host-os-modules/tree/main/sysctl) (recommended). Manual per-node steps are also provided.

#### (a) All clusters: apply `ptrace_scope = 2` via `HostOSConfiguration` (recommended)

The [sysctl module](https://github.com/Mirantis/mosk-host-os-modules/tree/main/sysctl)  applies kernel parameters at runtime through the host OS configuration API and persists them in a dedicated sysctl drop-in file. Use a `machineSelector` that matches **all cluster nodes** (for example, a label applied to management and child-cluster machines).

```yaml
apiVersion: kaas.mirantis.com/v1alpha1
kind: HostOSConfiguration
metadata:
  name: cve-2026-46333-ptrace-scope
  namespace: default
spec:
  configs:
  - module: sysctl
    moduleVersion: 1.2.0
    values:
      filename: cve-2026-46333-ptrace-scope
      cleanup_before: true
      options:
        kernel.yama.ptrace_scope: "2"
      state: present
  machineSelector:
    matchLabels:
      your-custom-label: "true"
```

Adjust `machineSelector` to target all nodes in your environment (management and child-cluster machines). Refer to the [sysctl module documentation](https://github.com/Mirantis/mosk-host-os-modules/tree/main/sysctl) for supported Ansible versions per MOSK release and for rollback (`state: absent`).

#### (b) Manual per-node procedure

1. **Check the current value** (MOSK defaults are more permissive than `2`):

```shell
cat /proc/sys/kernel/yama/ptrace_scope
```

1. **Set `ptrace_scope` to `2` immediately:**

```shell
echo 2 | sudo tee /proc/sys/kernel/yama/ptrace_scope
```

1. **Persist across reboots:**

```shell
echo "kernel.yama.ptrace_scope = 2" | sudo tee /etc/sysctl.d/99-ptrace-scope.conf
sudo sysctl --system
```

**Operational note:** `ptrace_scope = 2` may prevent unprivileged users and non-root operators from attaching debuggers to unrelated processes on the node. Mirantis testing on MOSK reference deployments did not identify impact on supported product operations; customers who rely on host-level debugging of arbitrary processes should account for this restriction.
